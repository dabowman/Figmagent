# Fix: [TOOL-006] Type coercion for tool parameters

**Pattern**: type-coercion
**Priority**: P1
**Issue**: #90
**Estimated savings**: eliminates cascading parallel-batch error cancellations (8 errors from 2 root causes in session 1)

## Problem

Agents pass numeric parameters as strings — `"4"` for a radius, `"0.85"` for a color
channel — and the Zod schemas use plain `z.number()`, which rejects strings. When one
call in a parallel batch errors on this, the whole batch is cancelled.

## Fix

Change the scalar numeric parameters in the two creation/modification tools from
`z.number()` to `z.coerce.number()`. This coerces numeric strings (`"4"` → `4`) while
leaving real numbers untouched and still rejecting genuinely non-numeric input. `.min()`
/`.max()`/`.positive()` chains run after coercion (so range validation is preserved), and
`.optional()` short-circuits `undefined` before coercion (no `NaN`).

## Changes

Replace every `z.number()` with `z.coerce.number()` in both files. Audited: all
occurrences are scalar design parameters an agent may pass as strings — color channels
(r/g/b/a), dimensions (x/y/width/height), padding, spacing, corner radius, stroke weight,
and font size/weight. None are array lengths or programmatic counts, so coercion is
uniformly safe (`.min`/`.max`/`.positive` still validate post-coercion; `.optional`
short-circuits `undefined`).

- `src/figmagent_mcp/tools/apply.ts` — 18 occurrences
- `src/figmagent_mcp/tools/create.ts` — 17 occurrences
