# Implementation Plan: `find` Tool

## Overview

A unified search tool that complements `get`. Where `get` says "show me this node," `find` says "show me nodes that match this." Returns compact matches with ancestry context, auto-grouped by nearest component/frame ancestor. Subsumes `scan_nodes_by_types` and `scan_text_nodes`.

## Design Decisions

- **Output**: YAML (via existing `serializeYaml`) with `meta` + `groups`. No `defs` section ‚Äî match criteria carry context.
- **Detail**: Compact by default (id, name, type, match reason, path, nearestComponent). Agent uses `get` for full details.
- **Grouping**: Auto-group by nearest COMPONENT/COMPONENT_SET/FRAME ancestor. Flat matches within each group.
- **Variable/style matching**: Direct bindings only. Style indirection is out of scope.
- **Criteria combination**: AND across fields, OR within each field's array.
- **`excludeDefinitions`**: When searching by `componentId`, skip matches inside those component definitions. Default true.
- **Plugin command name**: `find` (wire protocol), `find` (MCP tool name).

## Architecture Notes

### Plugin-side traversal strategy

The traversal is a single recursive walk. Key state tracked during descent:

```
traverseState = {
  path: string[],              // ancestor names from scope root
  nearestGroup: {              // closest grouping ancestor
    id: string,
    name: string,
    type: string               // COMPONENT | COMPONENT_SET | FRAME
  },
  insideDefinition: boolean,   // true if inside a componentId search target
}
```

For `componentId` matching:
- Use `getMainComponentAsync()` (async, handles remote components)
- `getMainComponentAsync()` can return null ‚Äî deleted components, detached instances ‚Üí treat as no-match
- Check both `mc.id` AND `mc.parent.id` when `mc.parent.type === "COMPONENT_SET"` to match set-level IDs
- Only call `getMainComponentAsync()` when `componentId` criterion is active

For `variableId` matching, two binding locations:
1. `node.boundVariables[field]` ‚Äî scalar props. Can be `{id}` or `[{id}]`
2. `node.fills[i].boundVariables.color` / `node.strokes[i].boundVariables.color` ‚Äî paint props

For `styleId` matching, guard against `figma.mixed`:
- `typeof node.textStyleId === "string"` before comparing (mixed returns Symbol)

### Grouping logic

Every match gets assigned to its nearest ancestor that is a COMPONENT, COMPONENT_SET, or top-level FRAME (direct child of the page). If no such ancestor exists, matches go into an "(ungrouped)" bucket. Groups are sorted by name.

### YAML output format (MCP side)

The MCP tool receives JSON from the plugin and converts to YAML. This mirrors `get`'s pattern of plugin returning JSON ‚Üí MCP formatting as FSGN YAML. The `find` output is simpler (no `defs`, no nested node trees) but uses the same `serializeYaml` function.
